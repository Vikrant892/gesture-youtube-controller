Hands-Free Media Control

Abstract. We describe a lightweight desktop utility that maps hand gestures observed through a standard webcam to YouTube keyboard shortcuts, allowing playback to be controlled without physical input. The pipeline uses MediaPipe Hands to extract twenty-one three-dimensional landmarks per frame, from which a small set of geometric features (finger-extension flags, thumb orientation, wrist velocity) is derived. A rule-based classifier assigns each frame to one of seven gesture classes, and the corresponding key press is injected via PyAutoGUI. The approach targets accessibility for users with motor impairments and general hands-busy settings, and prioritises interpretability and zero model-training cost over classification flexibility.

1. Introduction

Conventional media playback on the web assumes continuous access to a keyboard or pointing device. For users with transient or permanent upper-limb motor impairment, and for any user whose hands are occupied — during cooking, laboratory work, or physiotherapy exercises — this assumption breaks down. The YouTube keyboard shortcuts (space for play/pause, left and right arrows for seek, m for mute, f for fullscreen) are well documented[2], which makes a gesture-to-keystroke bridge an attractive minimum-viable solution: the web client is unchanged, no browser extension is required, and the mapping is transparent to the user. The system reported here takes that route and is deliberately small in scope.

2. Method

Frames are captured from the default camera at 480p, flipped horizontally so that the user's motion is mirrored, and passed to the MediaPipe Hands solution[1], which returns twenty-one landmarks per detected hand. For each frame the classifier derives (i) a five-element boolean vector indicating whether each finger is extended, (ii) the signed vertical offset between the thumb tip and the thumb interphalangeal joint, and (iii) the horizontal velocity of the wrist landmark over a four-frame window. These features are sufficient for a compact rule table, and each is cheap to compute. The resulting class label is converted to a keystroke via PyAutoGUI[3], with a short debounce interval so that a single gesture does not produce repeat key events.

2.1 Gesture Vocabulary

Seven gestures are recognised. The vocabulary was chosen to minimise pair-wise confusion under the rule classifier: the distinguishing feature of each class is a single, easily thresholded geometric predicate, and adjacent classes do not share that predicate.

3. Implementation

The classifier body is reproduced below, with debounce and smoothing removed for clarity. landmarks[i] follows the canonical MediaPipe index, so landmarks[4] is the thumb tip and landmarks[3] is the thumb interphalangeal joint. A fresh dictionary is returned on each call and no mutable state is carried between frames, which makes the function trivial to unit-test against recorded landmark sequences.

def classify(landmarks, wrist_dx): fingers = count_extended_fingers(landmarks) if fingers == 5: return "open_palm" if fingers == 0: return "fist" if fingers == 1 and thumb_only(landmarks): if landmarks[4].y < landmarks[3].y: return "thumb_up" return "thumb_down" if fingers == 1 and index_only(landmarks): if wrist_dx < -VEL_TH: return "point_left" if wrist_dx > VEL_TH: return "point_right" if fingers == 2 and index_and_middle(landmarks): return "peace" return None

4. Evaluation

Performance was measured on a 2020 consumer laptop (Intel i5-1035G1, integrated graphics, no external accelerator) running MediaPipe Hands at its default complexity. End-to-end latency is the time from frame capture to keystroke dispatch, averaged over two hundred trials per gesture under indoor office lighting. The false-positive rate counts frames in which a gesture was reported when the ground-truth class was none.

5. Limitations

  • Requires consistent lighting; there is no infrared fallback and the detector degrades under strong side-lighting or low ambient light.
  • Single hand only; two-hand gestures, which would expand the vocabulary without adding rule complexity, are not implemented.
  • Latency climbs to roughly 80 ms at 720p camera input, versus under 50 ms at 480p, which is noticeable for seek actions.
  • No adaptive learning; the user cannot train custom gestures, and the thresholds are hand-tuned for a right-handed user facing the camera.
